Verified the Tushare `stock_basic` docs against the lightweight provider implementation.

- Official docs confirm the current request contract is fundamentally correct for `stock_basic`:
  - HTTP POST to `http://api.tushare.pro`
  - JSON body with `api_name`, `token`, `params`, and `fields`
  - `params.list_status` and optional `params.exchange` are valid for this endpoint
- Current selected fields also match documented output columns used by the repo normalization path:
  - `ts_code`, `symbol`, `name`, `area`, `industry`, `market`, `exchange`, `list_status`, `list_date`, `is_hs`
- The main repo docs had overstated prerequisites:
  - a token alone is not enough
  - Tushare currently documents `stock_basic` as requiring `2000` points
  - a `100`-point account is therefore unlikely to work for this stock-list refresh API
- Net conclusion:
  - request shape is not the likely blocker
  - inside this sandbox, network egress to `http://api.tushare.pro` is still blocked
  - outside the sandbox, the next most likely blocker is token permission level rather than payload format
